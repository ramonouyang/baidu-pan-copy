## 问题描述

### 现象
1. 转存任务执行到第170批时触发百度限流(errno=-62)，任务立即停止并保存断点
2. 用户需要手动点击"恢复"才能继续，但恢复后仍被限流
3. 前端显示"系统已自动重试多次，但百度仍在限流"（实际代码并未重试）
4. BDCLND verify 和 get_share_children 也存在同样问题

### 根因分析

**3处转存循环** (main.py) 收到 errno -62/-9 后直接 `return`，无重试逻辑：
- `run_lazy_transfer()` line 1368
- `_recovery_transfer_from_file_list()` line 1797
- `run_lazy_transfer()` (恢复版) line 1981

**baidu_api.py** 中 `transfer_files()` line 1654 也立即返回，注释写着"重试会加重限流"（错误判断）。

**BDCLND verify** (line 767) 和 **get_share_children** (line 1048) 同样无重试。

### 限流机制分析

- 百度限流是**时间窗口制**，不是请求计数制
- 限流持续 30-60+ 分钟，从最后一次请求算起
- 每次手动重试会重置冷却计时器
- 有退避的自动重试不会雪崩（5次/25分钟 vs 用户手动无间隔重试）

## 修复方案

### 方案A：API层阻塞重试（BDCLND verify + get_share_children）
- 5次重试，延迟 60→120→180→240→300s（总计约25分钟）
- 适用于被调用方函数，阻塞当前线程

### 方案B：后台轮询重试（转存循环）
- 遇到限流 → 任务状态变为 `rate_limited` → 每5分钟自动重试
- 最多重试12次（1小时），成功自动恢复 `running` 状态
- 前端显示琥珀色"限流等待中"徽章，轮询不中断
- 用户无需操作，关页面也继续（后台 daemon 线程）
